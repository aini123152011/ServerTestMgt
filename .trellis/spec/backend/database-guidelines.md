# Database Guidelines

> Database patterns and conventions for this project.

---

## Overview

- **ORM**: SQLAlchemy 2.0 async mode (asyncpg driver)
- **Migrations**: Alembic with async engine
- **Database**: PostgreSQL 16+
- **Sync access**: Celery workers use `psycopg2` sync sessions (separate from async app)

---

## Model Conventions

All models extend three base classes from `models/base.py`:

```python
class MyModel(Base, IDMixin, TimestampMixin):
    __tablename__ = "my_models"
```

- `IDMixin`: auto-increment integer `id` primary key
- `TimestampMixin`: `created_at` and `updated_at` with `server_default=func.now()`
- Use `JSONB` columns for flexible/semi-structured data (test configs, hardware info, tags)
- Use `Enum` columns with `str, enum.Enum` mixin for type-safe status fields

---

## Query Patterns

```python
# Always use async session from dependency injection
async def my_endpoint(db: AsyncSession = Depends(get_db)):
    result = await db.execute(select(Model).where(Model.id == id))
    item = result.scalar_one_or_none()
```

- Use `select()` + `await db.execute()`, never `db.query()`
- Use `selectinload` or `joinedload` for relationships (lazy loading not supported in async)
- Pagination: `PageResult[T]` generic from `schemas/common.py`

## Celery Worker DB Access

Celery workers run in separate processes and cannot use async sessions. Use sync SQLAlchemy:

```python
from sqlalchemy import create_engine
from sqlalchemy.orm import Session

sync_engine = create_engine(settings.DATABASE_URL.replace("+asyncpg", "+psycopg2"))

def my_celery_task(job_id: int):
    with Session(sync_engine) as session:
        job = session.get(TestJob, job_id)
        # ... do work ...
        session.commit()
```

## Migrations

All models must be imported in `alembic/env.py` AND `init_db.py` for table detection:

```python
# alembic/env.py
from app.models.user import User          # noqa: F401
from app.models.device import Device      # noqa: F401
# ... every model file
```

> **Gotcha**: Forgetting to add a model import here means Alembic won't generate migrations for it, and `init_db.py` won't create the table on startup.

---

## Naming Conventions

- Table names: plural `snake_case` (e.g., `users`, `test_jobs`, `firmware_jobs`)
- Column names: `snake_case`
- Foreign keys: `<related_table_singular>_id` (e.g., `device_id`, `user_id`)
- Indexes: auto-generated by SQLAlchemy, add `index=True` on frequently queried columns

---

## Common Mistakes

### Forgetting model imports in alembic/env.py and init_db.py
New models won't be detected for migrations or auto-creation. Always add the import when creating a new model file.

### Using lazy loading in async context
SQLAlchemy async mode does not support implicit lazy loading. Use `relationship("X", lazy="selectin")` or explicit `selectinload()` in queries.

### Mixing async and sync sessions
Celery workers must use sync sessions. The FastAPI app uses async sessions. Never pass an async session to a Celery task.
